# Неделя 2 - День 10. Управление контекстом: разные стратегии (без summary)

## Задание

Реализовать в агенте 3 разных стратегии управления контекстом и переключатель между ними.
При выборе стратегии сжатие (компрессия) отключается.

## Доработки кода

### `agents/jarvis.py`

1. **Новые атрибуты**:
   - `context_strategy` — текущая стратегия (None / "sliding_window" / "sticky_facts" / "branching")
   - `sticky_facts` — словарь ключевых фактов (ключ-значение)
   - `branches` — словарь веток диалога
   - `current_branch_id`, `_next_branch_id`, `checkpoint_index` — состояние ветвления

2. **Новые таблицы и колонки в БД**:
   - `branches` — таблица для хранения метаданных веток
   - `context_strategy` (TEXT) — колонка в `sessions`
   - `sticky_facts` (TEXT) — колонка в `sessions` (JSON)

3. **Новые методы**:
   - `set_strategy(strategy)` — переключение стратегии (автоматически отключает сжатие)
   - `_get_sliding_window_messages()` — возвращает только последние 5 сообщений
   - `_apply_sliding_window()` — обрезает историю в памяти и БД до 5 сообщений
   - `_get_sticky_facts_messages()` — возвращает факты + последние 5 сообщений
   - `_extract_facts_initial()` — извлекает факты из всей истории при первом включении
   - `_extract_facts()` — обновляет факты через LLM после каждого обмена
   - `_parse_and_store_facts()` — парсит JSON с фактами
   - `_init_branches()` — инициализирует ветвление
   - `_get_branching_messages()` — возвращает сообщения текущей ветки
   - `save_checkpoint()` — сохраняет checkpoint в текущей ветке
   - `create_branch(name)` — создаёт новую ветку от checkpoint
   - `list_branches()` — список всех веток
   - `switch_branch(branch_id)` — переключение между ветками
   - `_save_strategy_state()` — сохраняет стратегию и факты в БД

4. **Изменённые методы**:
   - `__init__` — загружает стратегию и sticky_facts из БД
   - `chat()` — выбирает метод формирования сообщений по стратегии; после ответа обновляет факты / обрезает окно
   - `create_session()` — сбрасывает состояние стратегий
   - `switch_session()` — загружает стратегию и sticky_facts
   - `reset_conversation()` — сбрасывает sticky_facts и ветки
   - `get_stats()` — показывает текущую стратегию, факты и ветки
   - `_init_db()` — создаёт таблицу branches и колонки context_strategy/sticky_facts

### `week-02/day-10/w02t05.py`

Создан CLI-скрипт с поддержкой новых команд:
- `/strategy [sliding_window|sticky_facts|branching|off]` — установить/показать стратегию
- `/checkpoint` — сохранить checkpoint (для branching)
- `/branch <name>` — создать ветку от checkpoint
- `/branches` — список веток
- `/switch-branch <id>` — переключиться на ветку
- Старые команды: `/new`, `/sessions`, `/switch`, `/delete`, `/trim`, `/reset`, `/compress-on`, `/compress-off`, `/stats`, `/quit`

После каждого ответа показывается:
- токены последнего запроса
- текущая стратегия и режим сжатия
- количество фактов (для sticky_facts)
- информация о ветках (для branching)

## Сценарий проверки

### 1. Подготовка

```bash
python week-02/day-10/w02t05.py
```

### 2. Тестирование стратегий (3 отдельных запуска или 3 сессии)

#### Стратегия 1: Sliding Window

```text
/strategy sliding_window
```

Отправьте вопросы про ТЗ для телеграм-бота по изучению английского (4 вопроса).
После каждого ответа проверяйте `/stats` — в истории должно быть не более 5 сообщений.

#### Стратегия 2: Sticky Facts

```text
/new Тест sticky_facts
/strategy sticky_facts
```

Отправьте те же 4 вопроса. После каждого ответа смотрите `/stats` — количество фактов будет расти.

#### Стратегия 3: Branching

```text
/new Тест branching
/strategy branching
```

1. Отправьте 2 вопроса
2. `/checkpoint`
3. `/branch <имя ветки>`
4. Отправьте вопрос в новой ветке
5. `/switch-branch <id>`
6. Отправьте другой вопрос в основной ветке
7. `/branches` — проверьте, что обе ветки существуют

## Результаты проверки

### Расход токенов

| Параметр | Sliding Window | Sticky Facts | Branching |
|---|---|---|---|
| Всего токенов в сессии | 3 851 | 5 728 | 11 228 |
| Prompt последнего запроса | 1 179 | 1 730 | 1 218 / 698 (ветки) |
| Сообщений в истории (финал) | 5 | 8 | 4 / 8 (ветки main / бюджет) |

**Важно для Branching:** 11 228 токенов — это сумма по обеим веткам (6 сообщений в «Бюджет и стек» + 4 в «main»). Каждая ветка использует свою изолированную историю, поэтому прямой пересчёт на «одну беседу» не применим.

**Sticky Facts** — наибольшее число токенов из-за дополнительных API-вызовов для обновления фактов (19 фактов к концу).

**Sliding Window** — наименьший расход (3 851), так как история постоянно обрезается до 5 сообщений.

### Качество ответов

| Критерий | Sliding Window | Sticky Facts | Branching |
|---|---|---|---|
| Стек технологий | ✅ Cloud.ru, GPU, бюджет учтены | ✅ Cloud.ru, GPU, бюджет с ценами | ✅ Cloud.ru, GPU, детальная смета (ветка «Бюджет») |
| Архитектура | ✅ Telegram→ASR→LLM→DB, рекомендации | ✅ Telegram→ASR→LLM→DB, компоненты | ✅ Telegram→Whisper→LLM→DB, схема + SQL-код (ветка «main») |
| Связность ответов | ✅ Помнит все требования | ✅ Помнит, детализирует фактами | ✅ В каждой ветке — полный контекст до checkpoint |
| Детализация | Высокая | Высокая | Высокая (разные ракурсы в разных ветках) |

**Ключевой результат для Branching:** диалог реально разошёлся на 2 независимые линии:

1. **Ветка «Бюджет и стек»** (вопрос 3 + сравнение Whisper vs Google) — ответ содержит детальный расчёт стоимости (3500–6000 руб/мес) и сравнение ASR-решений с таблицей.
2. **Ветка «main»** (вопрос 4 + выбор БД) — ответ содержит архитектурную схему, SQL-код создания таблиц и рекомендации по СУБД.

При этом агент в каждой ветке помнит, что речь о телеграм-боте для английского и требованиях (ASR + LLM), — потому что история до checkpoint общая.

### Стабильность

1. **Sliding Window** — стабилен на коротких диалогах (<=5 сообщений). На длинных диалогах первые сообщения теряются безвозвратно.
2. **Sticky Facts** — очень стабилен, факты не теряются. Минус: дополнительные API-вызовы (влияет на скорость) и рост количества фактов (могут дублироваться).
3. **Branching** — стабилен в рамках каждой ветки. Checkpoint сохраняет общий контекст. Требует ручного управления ветками.

### Выводы

| Характеристика | Sliding Window | Sticky Facts | Branching |
|---|---|---|---|
| Экономия токенов | **Высокая** (3.8k) | Средняя (5.7k) | Низкая (11.2k сумма, но ветки изолированы) |
| Качество ответов | Высокое | Высокое | Высокое |
| Сложность реализации | Низкая | Средняя | Высокая |
| Удобство для пользователя | Прозрачно | Прозрачно | Требует ручного управления (/checkpoint, /branch, /switch-branch) |
| Риск потери контекста | Средний | **Низкий** | Низкий (в рамках ветки) |
| Когда применять | Короткие диалоги, частая смена тем | Информационно-насыщенные диалоги | Параллельные линии обсуждения с общим началом |

**Рекомендации:**
- **Sliding Window** — для коротких диалогов (до 10 сообщений). Простота и минимальный расход токенов.
- **Sticky Facts** — для длинных или информационно-насыщенных диалогов. Факты аккумулируются, контекст не теряется.
- **Branching** — для сложных задач, где нужно обсудить несколько альтернатив от общей точки. Позволяет сравнить разные варианты, не смешивая обсуждения.
