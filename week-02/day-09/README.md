# Неделя 2 - День 9. Управление контекстом: сжатие истории

## Задание

**Реализуйте механизм управления контекстом:**
- храните последние N сообщений "как есть"
- остальное заменяйте summary (например каждые 10 сообщений)
- храните summary отдельно и подставляйте его в запрос вместо полной истории

**Сравните:**
- качество ответов без сжатия
- качество ответов со сжатием
- расход токенов до/после

**Результат:**
- Агент, который работает с компрессией истории и экономит токены

## Доработки кода

### `agents/jarvis.py`

1. **Добавлена таблица `compressed_summaries` в SQLite** — для персистентного хранения сжатых summary между запусками агента.

2. **Добавлены методы**:
   - `_load_compressed_summaries()` — загружает все summary для текущей сессии из БД
   - `_save_compressed_summary()` — сохраняет новое summary в БД
   - `_clear_compressed_summaries()` — очищает все summary для текущей сессии
   - `compress_now()` — принудительное сжатие истории (без отправки сообщения)

3. **Исправлен `chat()`**:
   - Теперь использует `get_compressed_messages()` для формирования запроса к API вместо сырой `conversation_history`
   - При наличии сжатых фрагментов они подставляются в запрос как system-сообщения с префиксом `[АРХИВ: ...]`
   - Последние N сообщений остаются "как есть" для сохранения контекста текущего разговора

4. **Обновлены методы**:
   - `__init__()` — загружает сжатые summary при старте
   - `switch_session()` — загружает summary для выбранной сессии
   - `reset_conversation()` — очищает summary при сбросе
   - `trim_context()` — очищает summary при обрезке
   - `compress_history()` — сохраняет summary в БД

5. **Параметр `compression_interval = 5`** — каждые 5 сообщений история автоматически сжимается (для удобства проверки).

6. **Переключение режима сжатия**:
   - Параметр `compression_enabled` в `__init__` (по умолчанию `True`)
   - Новый метод `get_raw_messages()` — возвращает сырую историю без сжатия
   - `chat()` выбирает `get_compressed_messages()` или `get_raw_messages()` в зависимости от флага
   - Методы: `enable_compression()`, `disable_compression()`, `toggle_compression()`
   - Колонка `compression_enabled` в таблице `sessions` — режим сохраняется между запусками

### `week-02/day-09/w02t04.py`

Создан CLI-скрипт с поддержкой команд:
- `/compress` — принудительное сжатие истории
- `/compress-on` — включить сжатие
- `/compress-off` — выключить сжатие
- `/compress-toggle` — переключить режим сжатия
- `/stats` — показывает статистику сжатия (сколько фрагментов, экономия токенов, режим сжатия)
- Остальные команды: `/new`, `/sessions`, `/switch`, `/delete`, `/trim`, `/reset`, `/quit`

После каждого ответа показывается:
- токены последнего запроса
- общее заполнение контекста в %
- текущий режим сжатия (вкл/выкл)
- количество сжатых фрагментов и общая экономия токенов

## Результаты проверки

### Расход токенов

| Параметр | Со сжатием | Без сжатия |
|---|---|---|
| Всего токенов (5 сообщений + 3 вопроса) | 9 569 | 10 396 |
| Prompt последнего вопроса (Q3) | 1 751 | 1 987 |
| Заполнение контекста (финал) | 23.9% | 26.0% |

**Экономия ~8% токенов** за 8 диалоговых циклов. Сжатие 1 фрагмента (5 сообщений): 322 → 129 токенов (экономия 59.9%, ~193 токена).

### Качество ответов

| Вопрос | Со сжатием | Без сжатия |
|---|---|---|
| Q1: Элларион, оружие, спутник | ✅ Все детали верны | ✅ Все детали верны |
| Q2: Связь бургомистр — Мордекай — Вирмигорт | ✅ «Драконий договор», версия о сговоре | ✅ «Драконий договор», сговор с элитой |
| Q3: Бригитта и её план | ✅ Полугномиха, спасена 50 лет назад, план — украсть ключ на балу | ✅ Полугномиха, обязана жизнью, план — украсть ключ на балу |

**Оба режима ответили на все вопросы корректно и с сопоставимой детализацией.**

### Выводы

1. **Сжатие работает корректно** — 5 сообщений сжаты в summary с экономией 59.9%, контекст восстановлен без потери ключевых деталей.
2. **Качество не пострадало** — все проверочные вопросы получили правильные ответы в обоих режимах. Сжатая версия не уступала полной.
3. **Экономия токенов есть, но скромная** — на коротком диалоге (8 циклов) разница ~8%. Эффект будет заметнее на длинных диалогах (20+ сообщений), где каждые следующие 5 сообщений будут сжиматься.
4. **Сжатие summary-запроса не учитывается в статистике** — `compress_history()` сам вызывает API и тратит токены на суммаризацию. Эти токены не вычитаются из «сэкономленных». Чистая экономия начинает преобладать только после нескольких циклов сжатия.

### Рекомендация

Сжатие контекста оправдано для длинных сессий (30+ сообщений). На коротких диалогах выгода минимальна, и можно работать без сжатия, сохраняя максимальную точность деталей.
