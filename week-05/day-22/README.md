# Неделя 5 — День 22. Первый RAG-запрос

## Задание

**Реализуйте функцию:**
- вопрос → поиск релевантных чанков → объединение с вопросом → запрос к LLM

**Сравните:**
- ответ модели без RAG
- ответ модели с RAG

**Усиление:**
- составьте мини-набор из 10 контрольных вопросов по вашей базе
- для каждого вопроса зафиксируйте:
  - ожидание (что должно быть в ответе)
  - какие источники должны быть использованы (если применимо)

**Результат:**
- Агент с двумя режимами (с RAG / без RAG) + 10 контрольных вопросов и сравнение качества

## Решение

### Концепция

В JarvisAgent добавлен **RAG-режим** — флаг `rag_enabled`, который хранится в сессии (БД). Когда режим включён, перед каждым запросом к LLM агент автоматически:
1. Ищет релевантные чанки через `ragger.search.search()` по вопросу пользователя
2. Инжектирует их как system-сообщение перед user message
3. Модель получает актуальный контекст из базы знаний проекта

Режимы **RAG** и **MCP** независимы и могут работать одновременно:
- RAG — автоматическая подгрузка контекста, модель не управляет ретривом
- MCP — модель сама решает, вызывать ли `search_context` как tool

### Изменённые файлы

| Файл | Что сделано |
|---|---|
| `agents/jarvis.py` | В `__init__` добавлен `self.rag_enabled` (читается из сессии). В `chat()` — блок RAG-инжекции: поиск чанков → system message с текстом чанков → вставка перед user message |
| `agents/jarvis_session.py` | Миграция БД: колонка `rag_enabled INTEGER DEFAULT 0`. Метод `_save_rag_state()`. Чтение/запись во всех session CRUD |
| `agents/jarvis_commands.py` | Команда `/rag [on\|off]` — переключение режима. Добавлена в `/help` и в `/context` |

### Как пользоваться

```bash
# Включить RAG
/rag on

# Выключить
/rag off

# Статус
/rag
```

Включи RAG и задай вопрос по документам проекта — ответ будет основан на реальных чанках из базы знаний.

### Архитектура RAG-инжекции

```
messages = [system_prompt, ...context messages..., user_message]
                           ↓
if rag_enabled:
    chunks = rag_search(user_input)
    messages.insert(1, system{chunks})  ← перед memory-блоками
```

Порядок system-сообщений в итоговом промпте:
1. `[0]` — системный промпт агента
2. `[1]` — RAG-контекст (если включён)
3. `[2]` — Memory-блоки (Profile + TaskContext + Invariants)
4. Последующие — история диалога
5. Последнее — user message

### Сценарий проверки

1. **Запусти сервер.** Выполни `python ./webui/app.py`. Дождись сообщения `Running on http://127.0.0.1:5000`.

2. **Открой WebUI.** Перейди в браузере на `http://127.0.0.1:5000`. Интерфейс должен загрузиться — поле ввода чата и история (если есть).

3. **Проверь начальный статус RAG.** Отправь в чат: `/rag`. Ожидаемый результат: агент отвечает чем-то вроде *«RAG выключен (off)»* или *«RAG: выкл»*.

4. **Включи RAG.** Отправь: `/rag on`. Ожидаемый результат: агент подтверждает включение, например *«RAG включён»*.

5. **Подтверди статус через `/context`.** Отправь: `/context`. Ожидаемый результат: в ответе присутствует строка *«RAG: вкл»* (или *«RAG: on»*) рядом с остальными статусами (MCP, режим).

6. **Задай вопрос с RAG.** Отправь (или скопируй точный текст): `Какие стратегии чанкинга используются в проекте?` Ожидаемый результат:
   - Ответ содержит конкретные названия стратегий: *«structural»* и *«fixed»* (или *«структурный»* и *«фиксированный»*).
   - Упоминаются параметры (размер чанка, overlap), классы (например, `StructuralChunker`, `FixedSizeChunker`) или файлы (`ragger/chunking/`).
   - Ответ привязан к коду проекта, а не общие рассуждения о RAG.

7. **Выключи RAG.** Отправь: `/rag off`. Ожидаемый результат: подтверждение *«RAG выключен»*.

8. **Задай тот же вопрос без RAG.** Отправь: `Какие стратегии чанкинга используются в проекте?` Ожидаемый результат:
   - Ответ стал общим/абстрактным: перечисляются универсальные подходы (например, *«рекурсивный character split», «token-based», «semantic chunking»*).
   - Нет отсылок к конкретным файлам проекта, нет упоминаний `StructuralChunker`, `FixedSizeChunker` или параметров `chunk_size=1000`, `overlap=200`.
   - Ответ выглядит так, будто модель отвечает из своего общего знания, не имея доступа к документации проекта.

9. **Сравни ответы визуально.** Покажи на видео (или в тексте) оба ответа рядом:
   - **С RAG** — конкретный, со ссылками на код и параметры проекта.
   - **Без RAG** — общий, энциклопедичный, без привязки к проекту.
   - Комментарий: RAG превращает generic-ответ LLM в ответ, основанный на реальной базе знаний.

10. **Прогони контрольные вопросы (Усиление).** Включи RAG (`/rag on`) и по очереди задай 10 вопросов из таблицы ниже. Для каждого:
    - Убедись, что ответ ссылается на реальные детали проекта (имена классов, файлы, параметры).
    - Зафиксируй на видео 2–3 показательных примера.
    - Затем выключи RAG (`/rag off`), задай тот же вопрос — покажи контраст.

    **Рекомендуемые вопросы для видео:**
    - *«Сколько таблиц в БД Jarvis и какие?»* — с RAG назовёт 5 таблиц с именами; без RAG — общие слова про SQLite.
    - *«Какие этапы включает State Machine в Jarvis?»* — с RAG: PLANNING → EXECUTION → VALIDATION → DONE; без RAG — абстрактные этапы.
    - *«Какие модели LLM доступны через Cloud.ru API?»* — с RAG: конкретные Qwen/MiniMax; без RAG — общий список популярных моделей.
