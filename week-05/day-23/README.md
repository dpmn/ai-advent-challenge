- `agents/jarvis_commands.py` — `/rag` расширена: `config <key> <val>`, `compare <query>`. Добавлены `_handle_rag_config()` и `_handle_rag_compare()`
- `mcp_servers/ragger/server.py` — `_load_index` → `load_index`

### Команды:
```
/rag                    — статус + параметры
/rag on|off             — вкл/выкл RAG
/rag config <key> <val> — threshold / top_k_before / top_k_after / mode
/rag compare <query>    — сравнение threshold / rerank / hybrid
```

### Режимы:
- `threshold` — FAISS search → отсев по similarity score
- `rerank` — FAISS search → LLM-реранкинг всех чанков
- `hybrid` — FAISS search → threshold → LLM-реранкинг оставшихся

## Сценарий проверки

**Ключевой вопрос для сравнения:** 
```
Как устроен агент Jarvis: из каких модулей состоит и как они взаимодействуют?
```

Вопрос выбран потому, что:
- в индексе есть все 9 файлов `agents/*.py` — jarvis, commands, session, memory, context, compression, state_machine, invariants, mcp_manager
- FAISS найдёт много чанков с разными score (от 0.2 до 0.4)
- разные threshold и режимы дают разную полноту/глубину ответа — хорошо видно на видео

1. **Открой WebUI** — http://127.0.0.1:5000
   - Убедись, что интерфейс загрузился.

2. **Проверь статус и включи RAG**
   - `/rag` → статус (off, параметры по умолчанию)
   - `/rag on` → подтверждение
   - `/rag` → `enabled: true`

3. **Режим threshold, жёсткий порог (0.35)**
   - `/rag config mode threshold`
   - `/rag config threshold 0.35`
   - `/rag config top_k_before 10`
   - `/rag config top_k_after 3`
   - Задай вопрос: `Как устроен агент Jarvis: из каких модулей состоит и как они взаимодействуют?`
   - **Ожидаем:** ответ неполный — при threshold=0.35 большинство чанков отсекается (FAISS scores 0.2-0.4), в top_k_after=3 попадает 1-2 модуля. Агент видит мало контекста.

4. **Режим threshold, мягкий порог (0.15)**
   - `/rag config threshold 0.15`
   - Тот же вопрос
   - **Ожидаем:** ответ полнее — через порог проходит больше чанков, упоминается 4-6 модулей. Видно, что снижение threshold расширяет контекст.

5. **Режим rerank (LLM переоценивает все чанки)**
   - `/rag config mode rerank`
   - `/rag config top_k_after 3` (те же 3 чанка на выходе)
   - Тот же вопрос
   - **Ожидаем:** LLM сама выбирает 3 самых релевантных чанка из 10. Ответ может отличаться от threshold-режимов — LLM может выбрать чанк с низким FAISS-score (~0.2), но высоким семантическим соответствием. Качество ответа выше, но запрос к LLM дольше (~1-2 сек).

6. **Режим hybrid (threshold + rerank)**
   - `/rag config mode hybrid`
   - `/rag config threshold 0.15`
   - `/rag config top_k_after 5`
   - Тот же вопрос
   - **Ожидаем:** лучший результат — threshold (0.15) убирает совсем нерелевантное, rerank дооценивает оставшиеся, top_k_after=5 даёт максимум контекста. Ответ наиболее полный и точный, упомянуты почти все модули.

7. **Выключи RAG — контраст**
   - `/rag off`
   - Тот же вопрос
   - **Ожидаем:** ответ без документов — агент отвечает из своих знаний, менее детально или с оговорками. Наглядно видна разница с RAG-ответами.

## Выводы

### Сравнение режимов
- **threshold 0.35 (жёсткий порог):** Ответ перечислил 4 модуля (jarvis.py, mcp_manager.py, servers.json, webui/), взаимодействие описано обще — инициализация → запрос → команды → сохранение. Качество среднее: малая полнота, есть неточность (webui использует http.server вместо Flask). Порог 0.35 отсек бóльшую часть релевантных чанков (FAISS-scores 0.2–0.4).
- **threshold 0.15 (мягкий порог):** Ответ качественнее — таблица из 5 модулей (jarvis, mcpmanager, servers.json, webui/app.py, webui/*), детальное описание tool-calling loop, отмечены архитектурные принципы (stateful, no external libs, слоистый контекст). Снижение порога существенно расширило контекст.
- **rerank (LLM-переоценка, top_k_after=3):** Ответ отличный, несмотря на всего 3 выходных чанка. 8 модулей (jarvis, jarvis_session, jarvis_commands, jarvis_context, invariants, state_machine, mcp_manager, ragger/), полный lifecycle из 7 шагов, архитектурные принципы (синхронность, конфиг через JSON, изоляция MCP и RAG). LLM-реранкинг выбрал самые информативные чанки, отсеяв шум.
- **hybrid (threshold=0.15 + rerank, top_k_after=5):** Наилучший результат. 8 модулей с таблицей ключевых функций, полный lifecycle (7 шагов), раздел про инварианты и сохранение сессии, архитектурные принципы. Комбинация threshold + rerank дала максимальную полноту и точность.
- **RAG off (без документов):** Ответ непригоден — LLM не знает реальную архитектуру проекта. Говорит про «вымышленного ИИ-ассистента из вселенной Marvel», предлагает гипотетическую архитектуру. Аналогично в другой сессии без RAG.

### Что показало тестирование
- **Порог threshold напрямую влияет на полноту:** 0.35 отсекает почти всё (FAISS-scores чанков 0.2–0.4), 0.15 пропускает достаточно для адекватного ответа. Значение по умолчанию (0.2) — разумный компромисс.
- **LLM-реранкинг эффективнее threshold-фильтрации:** при одинаковом top_k_after=3 rerank дал значительно более полный ответ (8 модулей против 4–5 у threshold). Реренкинг выбирает семантически лучшие чанки, а не просто с высоким FAISS-score.
- **Hybrid — лучший режим:** threshold отсекает явный шум, rerank дооценивает оставшееся, top_k_after=5 даёт максимум релевантного контекста. Логичная комбинация, которая себя оправдала.
- **Без RAG ответ бесполезен** для вопросов о внутренней архитектуре проекта. Контраст разительный — от «Marvel Jarvis» до детального описания 8 модулей с таблицами и lifecycle.

### Замечания
- **Неточность в threshold-режиме:** агент утверждает, что webui использует http.server вместо Flask. Причина — чанк мог не содержать полной информации (обрезка). При threshold 0.35 в top_k_after=3 мог попасть частичный чанк без указания Flask. С более низким порогом или rerank-режимом эта неточность исчезает.
- **Размер top_k_after=3 узок для threshold:** при жёстком пороге в 3 чанка помещается мало информации. Рекомендуется для threshold ставить top_k_after=5 или больше.
- **Нет метрик времени:** тестирование не замеряло latency каждого режима. Реренкинг добавляет 1–2 сек к ответу — это приемлемо, но для production стоит учитывать.
- **Нет теста на перекрёстных вопросах:** проверка на одном вопросе показательна, но не гарантирует, что режимы работают так же стабильно на других типах запросов (фактологический, procedural, итп).
