# Неделя 4. MCP (Model Context Protocol)

**MCP (Model Context Protocol)** — открытый стандарт, разработанный для обеспечения взаимодействия LLM (нейросетей) с внешними сервисами, такими как корпоративные базы знаний, таск-трекеры, чаты или личные календари.


## 1. Что такое MCP и зачем он нужен

- **Стандарт, а не фреймворк:** MCP — это единый протокол, реализованный на разных языках (Python, Kotlin и др.), который позволяет любой модели (Claude, GPT и др.) общаться с любым внешним сервисом через унифицированный интерфейс.
- **Решение проблемы разрозненных API:** Раньше для каждого сервиса нужно было писать уникальную интеграцию, учитывая разные форматы ответов и кодов ошибок. MCP сводит эти различия к общему стандарту, работающему поверх HTTP.
- **Доступ к уникальным знаниям:** Протокол позволяет нейросетям выходить за рамки общедоступных обучающих данных и работать с закрытой информацией компании: чатами, документацией, бухгалтерией (например, 1С).


## 2. Преимущества и недостатки

- **Плюсы:**
    - **Прямой доступ из AI:** Можно просить модель запланировать встречу или проанализировать задачи в трекере без ручного копирования данных.
    - **Изоляция доступа (Tools):** Вы можете ограничить возможности агента, предоставив ему только конкретные инструменты (например, «только чтение календаря» без права создания встреч).
    - **Экономия времени:** Единый стандарт значительно ускоряет разработку AI-инструментов.
- **Минусы:**
    - **Безопасность:** Существует риск «логических бомб», когда через текстовый запрос модель может случайно выполнить опасную команду в консоли (например, `rm -rf`).
    - **Необходимость сервера:** Для работы требуется написание и поддержка MCP-сервера, что не всегда оправдано для простых локальных задач.


## 3. Сравнение эффективности: MCP против «Skills + CLI»

Критическая проблема MCP — **высокий расход токенов**:

- **Оверхед на обслуживание:** При каждом запросе модель вынуждена получать полный список доступных инструментов (схему), что может занимать до 98% всего контекста, оставляя лишь 2% на полезную работу.
- **Квадратичный рост:** Объем передаваемых данных растет квадратично по мере увеличения истории диалога и сложности схемы инструментов.
- **Альтернатива:** Использование связки **«Skill + CLI»** (командная строка) оказывается значительно дешевле и эффективнее, особенно при выполнении большого количества последовательных действий. Однако MCP остается актуальным для работы с удаленными серверами, где нет прямого доступа к CLI.


## 4. Пример реализации MCP-сервера на Python

```py
from mcp.server.fastmcp import FastMCP
import httpx

# 1. Создание инстанса сервера
mcp = FastMCP("WeatherService")

# 2. Описание инструмента (Tool) 
# Декоратор и описание (description) помогают LLM понять, когда вызывать функцию
@mcp.tool()
async def get_forecast(city: str) -> str:
    """
    Возвращает прогноз погоды для указанного города.
    Это поле 'description' используется для построения весов в запросе нейросети.
    """
    
    # 3. Маппинг и выполнение логики
    # В источнике это вызов HTTP-клиента к внешнему API
    async with httpx.AsyncClient() as client:
        # Инпут-схема (input schema) определяется аргументами функции
        response = await client.get(f"https://api.weather.example/v1/{city}")
        data = response.json()
        
        # 4. Возврат результата
        return f"Прогноз для {city}: {data['forecast']}"

if __name__ == "__main__":
    mcp.run()
```

- **Instance сервера:** Необходимо создать объект сервера, указав его название и версию.
- **Описание инструментов (Tools):** Для каждого инструмента обязательно указывается **имя** (name) и **описание** (description), чтобы модель могла соотнести запрос пользователя с нужной функцией.
- **Input Schema:** Вы четко определяете, какие параметры (например, название города или единицы измерения) агент должен передать для получения ответа

## Заметки

### Как ассистент узнаёт об инструментах

При каждом запросе `jarvis.py` собирает все подключённые MCP-серверы и преобразует их инструменты в OpenAI-формат 
`{"type": "function", "function": {"name": "...", "description": "...", "parameters": {...}}}` 
(см. [mcp_manager.py](../../agents/mcp_manager.py):344). 
Эти определения передаются в LLM как параметр **tools** на каждый вызов /chat/completions.
LLM видит: имя, описание, параметры — и решает, стоит ли вызывать инструмент.

```python
# jarvis.py:290-297
mcp_tools = self.mcp_manager.get_openai_tools()
response = self._call_api(messages, tools=mcp_tools or None)
```

### Как выстраивается пайплайн

Никакого жёсткого пайплайна нет. Это чистое **function calling** — LLM решает последовательность.

Механизм ([jarvis.py](../../agents/jarvis.py)):

1. LLM возвращает tool_calls — ассистент диспатчит через execute_tool()
2. Результат кладётся в историю как {"role": "tool", "content": "..."}
3. Ассистент снова вызывает LLM с обновлённой историей
4. LLM видит результат и может вызвать следующий инструмент
5. Цикл повторяется, пока LLM не перестанет вызывать инструменты (макс. 10 итераций)


То есть в Composer:

- fetch_data("apod") → LLM получает текст APOD
- LLM сама решает: «ага, теперь передам это в summarize_text»
- summarize_text(text, 30) → LLM получает сводку
- LLM решает: «теперь сохраню в файл»
- save_to_file("apod.txt", сводка)

Это не прописано в коде — это решение LLM на основе её понимания задачи и доступных инструментов.
