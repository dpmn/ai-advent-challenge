# Неделя 5. RAG (Retrieval-Augmented Generation)

**RAG (генерация, дополненная поиском)** — это паттерн, позволяющий дополнять ответы языковых моделей (LLM) знаниями из внешних или внутренних источников непосредственно во время работы (инференса). В отличие от базового обучения, RAG обращается к конкретным специфическим данным пользователя, а не только к общей базе знаний, на которой обучалась модель.

## Способы улучшения работы LLM

### 1. Контекстная оптимизация (Context Optimization)

Этот метод направлен на предоставление модели актуальных и специфических знаний в момент запроса.

- **Промпт-инжиниринг (Prompt Engineering):** прямое указание модели, как себя вести и какие данные использовать. Это делается через промпты, создание навыков (скиллов) и субагентов.
    - _Плюсы:_ это быстро и легко реализуемо (например, через MD-файлы).
    - _Минусы:_ данные не структурированы, и модель все равно может галлюцинировать.
- **RAG (Retrieval Augmented Generation):** паттерн дополнения ответов модели внешними знаниями непосредственно во время инференса. Система ищет подходящую информацию в вашей базе знаний и подкладывает её в контекст модели.
    - _Особенности:_ позволяет работать с корпоративными базами знаний, сохранять приватность (при использовании локальных моделей типа Ollama) и предоставлять ссылки на источники.

### 2. LLM-оптимизация (LLM Optimization)

Этот метод меняет внутренние параметры самой модели, чтобы она лучше справлялась с конкретными задачами.

- **Файнтюнинг (Fine-tuning):** дообучение модели на специфических данных, при котором меняются её **веса**.
    - _Когда использовать:_ если данных мало или они плохо размечены для RAG; если нужна сложная логика рассуждений или специфический стиль общения (вежливость, персонализация), который не решается простым поиском фактов

## Зачем нужен RAG?

Использование RAG решает несколько критических проблем современных LLM:

- **Ограниченность контекста:** Модели обучены на общих данных (Twitter, GitHub и т.д.), которые могут не иметь отношения к вашим частным данным.
- **Трудности поиска:** Поиск по простым файлам (например, MD-файлам) затруднен, если база знаний огромна.
- **Актуальность и точность:** Модель может путать версии документов или выдавать «ложноположительные» ответы — галлюцинации, которые выглядят правдоподобно, но являются фактически неверными.
- **Приватность:** RAG позволяет использовать локальные модели (например, через Ollama) для работы с закрытыми корпоративными данными без их передачи во внешние облака.

## Как это работает (технический процесс)

1. **Эмбеддинг-модели:** Текст переводится в **векторы** (массивы чисел), которые отражают семантический смысл слов.
2. **Векторный поиск:** Сравнение векторов происходит через **косинусное сходство**. Значение «1» означает полное совпадение смысла, «-1» — противоположность.
3. **Чанкинг и оверлап:** Большие тексты режутся на куски (**чанки**) по 500–1000 токенов. Чтобы не потерять смысл на границах разрезов, используется **оверлап** (наложение краев чанков друг на друга).
4. **Ретривал (извлечение):** При получении запроса система ищет наиболее подходящие чанки в базе данных и подмешивает их в промпт модели.
5. **Реранкинг:** Для повышения точности используется этап переранжирования, когда отдельная модель (cross-encoder) еще раз оценивает топ найденных результатов, чтобы выбрать наиболее релевантные.

### Пример реализации паттерна на Python

```python
import requests
import numpy as np

# Настройки
OLLAMA_URL = "http://localhost:11434/api/embeddings"
OLLAMA_GEN_URL = "http://localhost:11434/api/generate"
MODEL_NAME = "llama3" # или другая модель, загруженная в Ollama [4]

# 1. Данные (наша база знаний) [4]
knowledge_base = [
    "Политика Code Review: требуется минимум два апрува до мерджа. Максимальное время ревью 24 часа.",
    "SpaceX была основана Илоном Маском в 2002 году с целью снижения затрат на полеты в космос.",
    "Собаки не умеют играть в баскетбол, это негативный пример для тестов."
]

# Функция для получения эмбеддинга (вектора) через Ollama [2, 4]
def get_embedding(text):
    payload = {
        "model": "mxbai-embed-large", # пример специализированной модели [2]
        "prompt": text
    }
    response = requests.post(OLLAMA_URL, json=payload)
    return response.json()['embedding']

# Функция для расчета косинусного сходства [2, 6]
def cosine_similarity(v1, v2):
    return np.dot(v1, v2) / (np.linalg.norm(v1) * np.linalg.norm(v2))

# 2. Процесс индексации (превращаем базу знаний в векторы) [7, 8]
vector_store = []
for text in knowledge_base:
    vector_store.append({
        "text": text,
        "vector": get_embedding(text)
    })

# 3. Retrieval (Извлечение): Поиск подходящего чанка по запросу [3, 9]
query = "Какая у нас политика код-ревью?"
query_vector = get_embedding(query)

# Сравниваем запрос с каждым элементом в базе [6, 10]
best_chunk = None
max_sim = -1

for item in vector_store:
    sim = cosine_similarity(query_vector, item['vector'])
    if sim > max_sim:
        max_sim = sim
        best_chunk = item['text']

print(f"Найден релевантный контекст (схожесть {max_sim:.2f}): {best_chunk}")

# 4. Generation (Генерация): Подмешиваем чанк в промпт [1, 11]
full_prompt = f"""Используй только этот текст для ответа: {best_chunk}
Вопрос: {query}"""

payload_gen = {
    "model": MODEL_NAME,
    "prompt": full_prompt,
    "stream": False
}

response = requests.post(OLLAMA_GEN_URL, json=payload_gen)
print("\nОтвет ассистента:")
print(response.json()['response'])
```


## Размерность вектора

**Размер вектора** (или его размерность) — это количество чисел («параметров»), в которые модель преобразует фрагмент текста для описания его смысла.

Ключевые аспекты, касающиеся размеров векторов:

### 1. Стандартные размеры

Разные модели используют разную размерность векторов:

- **768** — распространенная размерность для многих базовых моделей.
- **1024** — более высокая размерность для повышения точности.
- **3072** — пример очень большой размерности для максимально детального семантического анализа.

### 2. Зачем нужно так много чисел?

Каждое число в векторе можно представить как координату в многомерном пространстве.

- **Аналогия с координатами:** Если в обычном мире у нас 3 измерения (X, Y, Z), то у эмбеддинг-модели их может быть **768 и более**. Это позволяет модели учитывать огромное количество нюансов значения слова или фразы.
- **Сравнение с классификацией:** Это похоже на описание объекта по множеству признаков (например, для вина это может быть год, сорт винограда, объем бутылки и т.д.). Чем больше параметров, тем точнее можно сравнить два объекта между собой.
- **Избегание «коллизий»:** Большое количество параметров работает как сильная хэш-функция — это минимизирует вероятность того, что разные по смыслу слова окажутся слишком близко друг к другу в векторном пространстве.

### 3. Точность vs Скорость

- **Выбор размерности:** Размер вектора напрямую влияет на точность поиска. Где-то нужно больше параметров для тонких различий, а где-то достаточно меньшего количества для экономии ресурсов.
- **Эффективность локальных моделей:** Модели с размерностью 768 или 3000 параметров считаются относительно легкими по сравнению с огромными LLM (у которых могут быть десятки миллиардов параметров). Это позволяет быстро выполнять векторизацию и поиск даже на локальном оборудовании, например, через **Ollama**.

### 4. Процесс формирования (Пулинг)

Размер итогового вектора предложения обычно является результатом **пулинга** — вычисления среднего арифметического векторов всех отдельных слов, входящих в это предложение. Это позволяет получить один компактный вектор фиксированной размерности (например, те же 768 чисел), который описывает смысл всего высказывания целиком


## Оптимизация и инструменты

- **Ollama:** Рекомендуется как одно из лучших решений для локального запуска эмбеддинг-моделей благодаря безопасности и совместимости.
- **Метрики качества:** Эффективность RAG оценивается по релевантности найденных чанков и «правдивости» (основан ли ответ на контексте или модель выдумала его из головы).
- **Комбинированный подход:** Для лучших результатов RAG стоит сочетать с промпт-инжинирингом и файн-тюнингом (дообучением весов модели).

## Ограничения

RAG — это не универсальное решение. Он **не поможет**, если:

- Данных изначально мало или они плохо структурированы.
- Требуется сложная логика рассуждений (это задача промпт-инжиниринга).
- Нужно изменить стиль общения или персонализацию агента (это решается настройкой агентов и тюнингом)

## Цели на неделю

- **Изучение работы с эмбедерами:** научиться использовать специальные модели, которые превращают текст в векторы для семантического поиска.
- **Реализация поиска по базе знаний:** создание механизмов, позволяющих находить релевантную информацию во внутренних данных.
- **Интеграция поиска в работу агента:** сделать так, чтобы агент мог самостоятельно обращаться к базе знаний во время инференса.
- **Подключение ссылок:** настроить систему так, чтобы ответы агента содержали ссылки на конкретные источники из вашей базы знаний (например, на статьи в Confluence или файлы в GitHub),.
- **Проведение реранкинга (Reranking):** в рамках практических заданий планируется научиться использовать отдельные модели (cross-encoders) для более точного отбора найденных фрагментов информации (чанков)
