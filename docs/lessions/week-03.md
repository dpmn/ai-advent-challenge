# Неделя 3. Память и состояние

О видах памяти агентов поговорили на той неделе: [week-02.md](week-02.md).

На этой неделе поговорим о памяти в контексте **выстраивания сложной агентской истории**, которую можно внедрить в разные места своего приложения.

## Группы агентов

### STATEFUL

Обладают состоянием, помнят этапы выполнения задачи, хранят контекст в слоях и знают свои ограничения. Они адаптируются под конкретного пользователя и контролируют переходы между стадиями работы.

**Особенности:**
- Помнит этап задачи
- Хранит контекст в слоях
- Знает ограничения
- Контролирует переходы
- Адаптируется под юзера

Профиль + состояние + инварианты + контроль переходов = управляемый агент.

### STATELESS

Работают с каждым запросом «с нуля», забывают глобальный контекст задачи, не имеют жестких правил и ограничений. Они выдают общие ответы для всех пользователей, не учитывая специфику проекта (например, могут предложить код на Python, хотя проект пишется на Kotlin).

**Особенности:**
- Каждый запрос с нуля - не долговременной истории
- Забывает контекст задачи
- Нет правил и ограничений
- Перепрыгивает этапы
- Один ответ для всех

Нет профиля -> неправильный стек
Нет состояния -> пропущены этапы
Нет инвариантов -> нет контроля


## Ключевые компоненты качественного агента

### 1. Персонализация:

Заточка агента под конкретного пользователя через определение его **стиля** (краткость, тон), **ограничений** (constraints) и **контекста** (цели использования). Это часто реализуется через предварительное «интервью» с пользователем, данные которого сохраняются (например, в MD-файлы или БД) и подмешиваются в системный промт.

Всегда нужно подбирать способ и место хранения файла персонализаций исходя из задачи.
Также нужно понимать как всё организовать в один общий пайплайн. Например, в системные промпт можем подгружать system + profile + query.


### 2. State Machine (Машина состояний)

Основа хорошего агента, которая вносит детерминизм в работу нейросетей. Базовая структура включает четыре этапа: **Planning** (собираем требования, утверждаем план), **Execution** (пишем код, создаём артефакты), **Validation** (тесты, ревью, соответствие плану) и **Done** (задача завершена, фиксируем результат). Жёсткое управление переходами между ними в коде программы помогает избежать галлюцинаций.

**Пример реализации на Python:**

```python
from enum import Enum
from typing import Dict, List, Optional

# Определение состояний как данных
class AgentState(Enum):
    PLANNING = "PLANNING"
    EXECUTION = "EXECUTION"
    VALIDATION = "VALIDATION"
    DONE = "DONE"

# Карта разрешенных переходов (Transitions Map)
# Это делает логику агента детерминированной [1, 2]
ALLOWED_TRANSITIONS: Dict[AgentState, List[AgentState]] = {
    AgentState.PLANNING: [AgentState.EXECUTION],
    AgentState.EXECUTION: [AgentState.VALIDATION, AgentState.PLANNING],
    AgentState.VALIDATION: [AgentState.DONE, AgentState.EXECUTION],
    AgentState.DONE: []
}

class StatefulAgent:
    def __init__(self, user_id: str):
        self.user_id = user_id
        # Текущее состояние хранится как данные, которые можно сохранить в БД или файл [5, 6]
        self.current_state = AgentState.PLANNING
        self.context = []
        self.plan = None

    def transit_to(self, next_state: AgentState):
        """Контролирует переходы между стадиями [1, 7]."""
        if next_state in ALLOWED_TRANSITIONS[self.current_state]:
            print(f"Переход из {self.current_state.value} в {next_state.value}")
            self.current_state = next_state
        else:
            print(f"Ошибка: переход из {self.current_state.value} в {next_state.value} запрещен!")

    def build_prompt(self, user_query: str) -> str:
        """Инжектирует текущее состояние и контекст в промт [8]."""
        # Здесь данные о состоянии превращаются в инструкции для LLM
        prompt = f"""
        Ты находишься в стадии: {self.current_state.value}.
        Текущий план: {self.plan}
        Инварианты: Использовать только Python и библиотеку FastAPI.
        
        Задача пользователя: {user_query}
        
        ДЕЙСТВУЙ ТОЛЬКО В РАМКАХ ТЕКУЩЕЙ СТАДИИ. Не перепрыгивай этапы [8].
        """
        return prompt

# Пример работы:
agent = StatefulAgent(user_id="user_123")
# Агент начинает в PLANNING, пробуем перейти в EXECUTION
agent.transit_to(AgentState.EXECUTION) # Успешно
# Пробуем перепрыгнуть стадию (из EXECUTION сразу в DONE)
agent.transit_to(AgentState.DONE)      # Ошибка: переход запрещен [1]
```

### 3. Инварианты

Неизменные ограничения, которые агент не должен нарушать независимо от вопроса пользователя (например: определенный стек технологий, архитектура MVI, запрет на использование конкретных библиотек типа RX Java).

Инварианты можно хранить отдельно в MD-файлах или базе данных и подгружать их в зависимости от конкретной задачи или профиля пользователя

В отличие от текстовых инструкций в промте, которые модель может игнорировать из-за их вероятностной природы, проверка в коде обеспечивает «железное» соблюдение правил.

**Пример реализации на Python:**

```python
from abc import ABC, abstractmethod
from typing import List, Optional

# 1. Базовый класс для инвариантов [1]
class Invariant(ABC):
    @abstractmethod
    def check(self, response: str) -> bool:
        """Возвращает True, если нарушений нет, и False в противном случае."""
        pass

    @abstractmethod
    def get_error_message(self) -> str:
        """Сообщение об ошибке для повторного промта нейросети."""
        pass

# 2. Конкретные реализации ограничений [5, 6]
class ForbiddenLibraryInvariant(Invariant):
    def __init__(self, forbidden_libs: List[str]):
        self.forbidden_libs = forbidden_libs

    def check(self, response: str) -> bool:
        # Проверяем вхождение запрещенных строк в ответе
        return not any(lib in response for lib in self.forbidden_libs)

    def get_error_message(self) -> str:
        return f"Ошибка: Использование библиотек {self.forbidden_libs} запрещено!"

class TechStackInvariant(Invariant):
    def check(self, response: str) -> bool:
        # Например, проект только на Kotlin [5]
        return "fun main" in response or "package" in response

    def get_error_message(self) -> str:
        return "Ошибка: Ответ должен быть написан на языке Kotlin."

# 3. Интеграция в пайплайн агента [4]
class AgentValidator:
    def __init__(self, invariants: List[Invariant]):
        self.invariants = invariants

    def validate_response(self, response: str) -> Optional[str]:
        """Проверяет ответ по всем инвариантам и возвращает первую ошибку."""
        for invariant in self.invariants:
            if not invariant.check(response):
                return invariant.get_error_message()
        return None

# Пример логики работы в цикле
invariants = [
    ForbiddenLibraryInvariant(["RX Java", "AsyncTask"]),
    TechStackInvariant()
]
validator = AgentValidator(invariants)

llm_response = "Вот твой код на Java с использованием RX Java..." # Гипотетическая галлюцинация [7]
violation = validator.validate_response(llm_response)

if violation:
    # Если зафиксировано нарушение, отправляем на ретрай [1, 4]
    print(f"Повторный запрос к LLM: {violation}")
else:
    print("Ответ прошел валидацию.")
```


## Антипаттерны

### 1. «Всё в одном промте» (Everything in one prompt)

Попытка при каждом запросе отправлять нейросети абсолютно всю имеющуюся информацию: полную историю диалога, все профили пользователя, все инварианты и правила. Это плохая практика, так как она приводит к **быстрому переполнению контекстного окна**. Рекомендуетcя делить данные на слои и добавлять в промт только то, что необходимо для конкретной задачи (например, если в данный момент профиль или история не важны, их стоит пропустить).

### 2. Отсутствие валидации переходов между состояниями

LLM по своей природе стремится «угодить» пользователю и может попытаться перепрыгнуть через этапы или выполнить действие, к которому система еще не готова. Если не контролировать переходы программно (в коде), агент может нарушить логику работы стейт-машины, что приведет к ошибкам в выполнении сложных задач.

### 3. Использование только текстовых инвариантов

Полагаться исключительно на инструкции внутри промта для задания жестких ограничений (например, «никогда не используй Java») — это антипаттерн. Промпт — это всего лишь инструкция, которую модель может нарушить из-за своей вероятностной природы.


## Архитектура и реализация

- **Пайплайн работы:** Перед каждым взаимодействием из хранилища (БД, API, MD-файлы) загружаются профиль пользователя, текущее состояние задачи и инварианты. Эти данные объединяются в промт-билдере и отправляются в LLM.
- **Валидация:** Ответ модели должен фильтроваться на соответствие инвариантам. Если правила нарушены, агент должен получить команду на переделку.
- **Изоляция:** Для максимальной надежности стадии можно разделять на уровне программного кода, запуская для каждой задачи отдельные сессии или функции.


## Практическая польза

Внедрение такой архитектуры (стейт-машины, инвариантов и персонализации) повышает эффективность работы код-ассистентов и сервисов в десятки и сотни раз. Это позволяет агенту корректно продолжать работу даже после долгого перерыва, восстанавливая весь контекст из сохраненного состояния.
