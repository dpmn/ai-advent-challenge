# Неделя 3 - День 11. Модель памяти ассистента

## Задание

**Описать и реализовать модель памяти для ассистента.** 
Разумеется, все правила управления контекстом (из прошлой недели) применимы и здесь.

**Нужно разделить информацию на три типа:**
1. Краткосрочная (текущий диалог) [_как понимаю, это у нас уже реализовано в рамках сессии_]
2. Рабочая (данные текущей задачи), TaskContext
3. Долговременная (профиль, решения, знания) [_простой вариант - просто предоставляем возможность загрузки/редактирования .md_]

**Сделать так, чтобы:**
- разные типы памяти хранились отдельно
- можно было выбирать что и куда сохраняется

**Проверить:**

Для проверки давай возьмём домен: агент для превращения текстовых запросов в sql для аналитики.

За идею можно взять этот проект, но **максимально упростив** его (не забывай, что выполняем учебное задание): https://github.com/agno-agi/dash

Нужно посмотреть:
- какие данные попадают в каждый слой
- как это влияет на ответы ассистента

### Архитектура трёхуровневой памяти

```
┌─────────────────────────────────────────────────────────────┐
│  SYSTEM PROMPT (собирается перед каждым запросом)           │
│                                                             │
│  [system] Базовый системный промпт                          │
│  [system] 👤 Профиль: dbms=PostgreSQL, sql_style=...        │  ← Long-term
│  [system] 📋 TaskContext: database=sales_analytics, ...     │  ← Working
│  [user]   Текущий запрос пользователя                       │
│  ...       История диалога (Short-term)                     │  ← Short-term
└─────────────────────────────────────────────────────────────┘
```

### Хранение слоёв

| Слой | Класс | Хранилище | Команды |
|------|-------|-----------|---------|
| Short-term | `conversation_history` | SQLite (`messages`), память | — (сессия) |
| Working | `TaskContext` | Память (`self.task_context`) + SQLite (`sessions.task_context`, JSON) | `/task`, `/task clear` |
| Long-term | `Profile` | Markdown-файлы (`agents/memory/profiles/<name>.md`) | `/profile`, `/profile set`, `/profiles` |

## Сценарий для видео

Запись через WebUI (http://127.0.0.1:5000). Демонстрируем, как три слоя памяти влияют на ответы SQL-агента.

### Подготовка к записи

1. Убедись, что `.env` содержит `CLOUDRU_SECRET_KEY`.
2. Запусти Flask: `python webui\app.py`
3. Открой в браузере: http://127.0.0.1:5000
4. Удали старые сессии через кнопку ✕, чтобы не захламлять экран.
   **Начать запись видео.**

### Шаг 1. Создать новую сессию

| Действие | Что ввести | Ожидаемый результат |
|----------|-----------|---------------------|
| Нажать «+ New Session» | Имя: `Text-to-SQL демо` | Сессия появилась в боковой панели |

### Шаг 2. Настроить долговременную память (Profile)

Вводим в чат команды:

| № | Команда | Что происходит |
|---|---------|----------------|
| 1 | `/profile set dbms PostgreSQL` | Задан тип БД |
| 2 | `/profile set naming_convention snake_case, lower_case_table_names` | Соглашение об именах |
| 3 | `/profile set sql_style WITH clauses для сложных запросов, явные JOIN, алиасы таблиц по первой букве` | Стиль SQL |
| 4 | `/profile set date_format YYYY-MM-DD` | Формат дат |

**Показать результат:**
| 5 | `/profile` | Выведутся все 4 поля профиля |

**Объясни на видео:** *«Долговременная память сохраняется в Markdown-файле `agents/memory/profiles/default.md`. Если перезапустить сервер — профиль останется.»*

### Шаг 3. Настроить рабочую память (TaskContext)

Вводим схему базы данных, с которой будет работать агент:

| № | Команда | Что происходит |
|---|---------|----------------|
| 1 | `/task database sales_analytics` | Задаём имя БД |
| 2 | `/task table.orders id (INT PK), customer_id (INT FK), order_date (DATE), total_amount (DECIMAL), status (TEXT)` | Описываем таблицу orders |
| 3 | `/task table.customers id (INT PK), name (TEXT), email (TEXT), registration_date (DATE), country (TEXT)` | Описываем таблицу customers |
| 4 | `/task table.order_items id (INT PK), order_id (INT FK), product_name (TEXT), quantity (INT), unit_price (DECIMAL)` | Описываем таблицу order_items |
| 5 | `/task current_task Подготовить дашборд продаж за текущий месяц` | Текущая цель |

**Показать результат:**
| 6 | `/task` | Выведутся все 5 полей TaskContext |

**Объясни на видео:** *«Рабочая память хранит данные текущей задачи. Она инжектируется в промпт перед каждым запросом.»*

### Шаг 5. Задать вопрос агенту — проверка влияния памяти

Вводим текстовый запрос (НЕ команду, а обычное сообщение):

| № | Запрос | Что проверяем |
|---|--------|---------------|
| 1 | `Покажи 10 самых активных клиентов по сумме заказов` | Агент должен сгенерировать PostgreSQL (см. Profile), с JOIN (см. Profile-CTE), используя таблицы `orders`, `customers` из TaskContext |

**Ожидаемый SQL (пример):**
```sql
WITH customer_spending AS (
    SELECT customer_id, SUM(total_amount) AS total
    FROM orders
    GROUP BY customer_id
)
SELECT c.name, cs.total
FROM customer_spending cs
JOIN customers c ON cs.customer_id = c.id
ORDER BY cs.total DESC
LIMIT 10;
```

**Покажи на видео:** *«Агент использовал CTE (как указано в профиле), явный JOIN, snake_case, и знает про колонки `total_amount`, `customer_id` из схемы в TaskContext.»*

**Объясни на видео:** *«Краткосрочная память — это история диалога внутри сессии. Агент помнит, что мы только что спрашивали про топ-10 клиентов, и понимает слово "то же самое".»*
